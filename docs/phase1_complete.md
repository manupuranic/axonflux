# AxonFlux — Project Memory Summary

**Phase 1: Raw Ingestion Complete**

---

## 1. Project Overview

- **What it is:** AxonFlux is a raw-first, append-only data platform for supermarket billing software that only provides manual CSV/XLS exports (no API).
- **Core problem:** Billing data is messy, inconsistent, and incomplete (no conversion events, spelling issues, snapshot vs event confusion). AxonFlux preserves raw truth first, then derives clean, auditable analytics, inventory, and future AI insights.

---

## 2. Current State

### ✅ Implemented & Working (Phase 1 Complete)

- **All raw ingestion scripts implemented and tested:**
  - Sales **item-wise**
  - Sales **bill-wise**
  - Purchase **item-wise**
  - Purchase **bill-wise**
  - Supplier master (**snapshot-based**)
  - Item combinations (**snapshot-based**)

- **Raw schemas frozen** and aligned exactly to billing exports (including UI/junk columns).

- **Unified ingestion engine implemented:**
  - Shared ingestion core
  - Per-table `HEADER_MAP` as schema contract
  - Per-report reader functions (CSV / Excel, `skiprows`, totals-row handling)

- **Performance validated and optimized:**
  - Chunked bulk inserts using engine-level connections
  - Single transaction per file
  - ORM sessions explicitly avoided for high-volume ingestion

- **Ingestion boundary guarantees:**
  - Pandas `NaN` / `NaT` → SQL `NULL`
  - Float-like IDs coerced to TEXT (e.g. `2796.0 → "2796"`)
  - UUIDs generated by the database, not Python

- GitHub repository structure stabilized; ingestion architecture documented.

## 3. Architecture & Stack

- **Language:** Python
- **Database:** PostgreSQL (Supabase)
- **ORM:** SQLAlchemy (2.0 style)
- **Ingestion:** Pandas → chunked bulk inserts (`executemany` semantics)

### DB Access Pattern

- ORM `Session` → reads, small writes, reconciliation logic
- Engine-level `Connection` → raw ingestion and historical backfills

### Storage Layers

- `raw.*` — append-only truth storage (events + snapshots)
- `recon.*` (planned) — reconciliation views
- `derived.*` (planned) — business-ready tables

## 5. Key Decisions & Invariants (Non-Negotiable)

- Raw tables are **append-only**
  - No updates
  - No deletes (except full `TRUNCATE` during development)

- Raw schema mirrors exports **exactly**; no inference or normalization.

- Events and snapshots are modeled separately:
  - **Events:** sales, purchases
  - **Snapshots:** supplier master, item combinations

- Inventory is **computed later**; snapshot stock values are sanity checks only.

- Conversions are **inferred via reconciliation**, never assumed.

- All missing values are stored as **SQL NULL**, not Pandas artifacts.

- Raw schema is **closed** unless billing exports change.

- Historical data **must be backfilled** (not optional).

---

## 6. Known Problems & Limitations

- Billing system does not export:
  - Stock conversion events
  - Stock adjustments

- Item naming is inconsistent; no canonical product layer yet.
- Supplier identities drift over time (expected; preserved in raw).
- Export automation is manual; scraping is a last resort.
- Raw layer intentionally does **not** resolve inconsistencies.
