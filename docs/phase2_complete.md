---

## 1. Project Overview

- **What it is:** LedgerAI is a raw-first, append-only data platform for supermarket billing software that provides only manual CSV/XLS exports (no API).
- **Core problem:** Billing data is messy, inconsistent, incomplete, and semantically overloaded (no conversion events, inconsistent units, snapshot vs event confusion).
- **Core philosophy:** Preserve raw truth first, then _explicitly_ derive interpretations through reconciliation and derived layers.

---

## 2. Current State (As of Now)

### ✅ Raw Ingestion (Phase 1 — Complete)

All raw ingestion scripts are implemented, tested, and stable:

- **Sales**
  - Item-wise
  - Bill-wise
- **Purchases**
  - Item-wise
  - Bill-wise
- **Snapshots**
  - Supplier master
  - Item combinations

**Raw ingestion guarantees:**

- Append-only (no updates, no deletes)
- Raw schemas mirror exports exactly (including UI/junk columns)
- Pandas `NaN` / `NaT` → SQL `NULL`
- Float-like IDs coerced to TEXT
- UUIDs generated by DB
- Multiple imports allowed for same logical records (late data supported)

**Performance & tooling:**

- Chunked bulk inserts
- Engine-level connections for ingestion
- ORM sessions reserved for reads / reconciliation
- Schema frozen unless billing exports change

**Historical backfill:**

- ~1 year of data ingested
- Month-by-month
- Late-arriving bills validated and appended via new import batches
- No raw data overwritten

### ✅ Reconciliation (Phase 2 — Complete)

#### 1. Sales Bill-wise vs Item-wise

- Structural mismatches investigated
- Rare bill number collision identified and contained
- Composite reconciliation identity used:
  - `(bill_no, bill_datetime_raw, operator_name, location_name)`
- Final state:
  - Only rounding deltas (< 1)
  - No structural data loss

#### 2. Purchase Bill-wise vs Item-wise

- Quantities reconcile exactly
- Monetary fields diverge systematically due to bill-level adjustments
- One isolated structural anomaly (bill with no item rows), documented and preserved

#### 3. Quantity Flow Sanity (Purchases → Sales)

- Result:
  - Sales quantity > purchase quantity over backfill window
- Interpretation:
  - Expected due to:
    - Unit mismatches (packs vs loose units)
    - Opening stock not captured
    - Free quantities
    - Self-packed items
- Conclusion:
  - Sanity check **PASSED**
  - Raw quantities are directionally sound but not inventory-accurate

### ✅ Derived Tables (Foundational)

#### `derived.daily_sales_summary`

- Grain: **1 row per day**
- Authoritative sources:
  - Revenue → sales bill-wise
  - Quantity → sales item-wise
- Strict date parsing applied
- Rows with unparsable dates excluded by design
- Cross-verified with billing UI

#### `derived.daily_purchase_summary`

- Grain: **1 row per day**
- Authoritative sources:
  - Monetary values → purchase bill-wise
  - Quantity → purchase item-wise
- Late-arriving bills appended to raw and reflected via rebuild
- Cross-verified with billing UI (including late data)

**Derived table invariants:**

- Fully rebuildable from raw
- Never manually patched
- No inventory guarantees implied

## 3. Architecture & Stack

- **Language:** Python
- **Database:** PostgreSQL (Supabase)
- **ORM:** SQLAlchemy 2.0
- **Ingestion:** Pandas → chunked bulk inserts
- **Date handling:** Raw as TEXT, parsed only in recon/derived layers

### Storage Layers

- `raw.*` — append-only event & snapshot storage
- `recon.*` — diagnostic and validation views
- `derived.*` — business-ready, opinionated interpretations

---

## 4. Operating Model (Critical)

- Raw data represents **what was known, when it was known**
- Late-arriving data is ingested as **new raw batches**
- Derived tables are **recomputed**, never manually edited
- Duplicates are tolerated in raw, resolved in derived logic if needed
- No silent correction of inconsistencies

---

## 5. Known Limitations (Explicit)

- No stock conversion events exported
- No stock adjustment events exported
- No opening stock snapshot at start of backfill
- Unit conversions (packs → units, weight normalization) not yet modeled
- Self-packed items not yet resolved

These are structural limitations, not ingestion errors.

---

## 6. What LedgerAI Can Safely Do Now

- Daily / weekly dashboards
- Revenue analytics
- Demand trend analysis
- Procurement vs demand comparisons
- ML feature generation (non-inventory)
- Supplier cadence analysis

---

## 7. What Is Explicitly Deferred

- Exact inventory computation
- Canonical product resolution
- Unit conversion modeling
- Forecasting that depends on stock accuracy

These require additional modeling layers.
