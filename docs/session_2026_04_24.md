## 1. Pipeline Reliability Fixes

### Encoding crash (`cp1252` → `utf-8`)
Playwright/Chromium outputs UTF-8. After reinstalling packages, `bcrypt>=4.0` and a fresh venv caused two crashes:

- **bcrypt init crash**: `passlib` calls `detect_wrap_bug()` at import time with a >72-byte secret; `bcrypt>=4.0` raises `ValueError` instead of silently truncating. Fix: pin `bcrypt<4.0.0` in `api/requirements.txt`.
- **Subprocess pipe encoding**: `subprocess.Popen(text=True)` uses the system default (`cp1252` on Windows). Playwright stdout has chars outside cp1252. Fix: add `encoding="utf-8", errors="replace"` to `_stream_proc` in `api/routers/pipeline.py`.

### Ingestion logs silent
`ingest_all.py` and `ingest_core.py` use `print()` without `flush=True`. When stdout is piped, Python buffers until process exit — so no live log appeared in the modal. Fix: pass `-u` (unbuffered) flag to all python subprocess calls in the pipeline runner.

### Unhandled exceptions swallowed
If any exception escaped the background task functions (`_run_full_refresh`, `_run_pipeline`), the DB run record stayed stuck at `status="running"` forever. Fix: wrap both functions in `try/except Exception`, emit the full traceback to `log_output`, and call `_finish_run(..., "failed")`.

## 2. Pipeline UX — Stop + Retry

**Problem**: no way to cancel a hung pipeline from the UI.

**Backend** (`api/routers/pipeline.py`):
- `_active_procs: dict[str, Popen]` — module-level dict keyed by `run_id`, updated inside `_stream_proc`.
- `POST /api/pipeline/{run_id}/cancel` — calls `proc.kill()` (on Windows = `TerminateProcess`, immediate), then pre-emptively marks DB status `"cancelled"`. Background thread updates it again when it exits; no race — both writes set the same value.

**Frontend** (`PipelineTriggerModal.tsx`):
- Red "Stop" button appears while `isRunning`.
- On cancel: optimistic reset to idle state so user can retry immediately without waiting for the thread to exit.
- Footer shows "Retry" + "Close" on `cancelled` status.

## 4. Supplier Master Data Bug Fix

**Bug**: `derived.supplier_location` queried all rows from all ingested batches with no deduplication. Every re-ingest of the supplier CSV multiplied rows in the view. Joins in step 06 and the suppliers API would fan out, producing duplicates or wrong region for products.

**Fix** (`sql/rebuild_derived/05_necessary_views.sql`):
```sql
-- Before (broken)
SELECT supplier_name_raw, city_raw FROM raw.raw_supplier_master;

-- After (correct — same pattern as latest_item_combinations)
SELECT DISTINCT ON (supplier_name_raw)
    supplier_name_raw AS supplier_name,
    CASE WHEN city_raw = 'Bellary' THEN 'BELLARY' ELSE 'OUTSIDE' END AS supplier_region
FROM raw.raw_supplier_master
ORDER BY supplier_name_raw, imported_at DESC;
```

Requires a pipeline rebuild (Rebuild Only) to take effect.

## 5. Product Health — Avg Monthly Consumption + Supplier Info

### Correct avg monthly consumption
Previous approach (`last_30_day_avg × 30`) was wrong — `last_30_day_avg` is a rolling daily average, not actual monthly units. Correct formula:

```sql
SELECT ROUND(SUM(quantity_sold)::numeric / 6, 2)
FROM derived.product_daily_metrics
WHERE product_id = :barcode
  AND date > CURRENT_DATE - INTERVAL '180 days'
```

Applied in: `GET /api/analytics/health-signals`, `GET /api/products/{barcode}`, both CSV exports.

### Supplier info on health signals
Added correlated subquery in `health-signals` endpoint:
```sql
SELECT STRING_AGG(DISTINCT supplier_name, ', ' ORDER BY supplier_name)
FROM derived.product_supplier_mapping
WHERE product_id = h.product_id
```

### Where it shows
- **Health table**: columns removed (too cramped).
- **Product Drawer** (slide-in from health table row): "Avg Monthly (units)" stat card + "Procured From" supplier badges.
- **Product Detail page** (`/products/{barcode}`): "Avg Monthly" as first stat card; "Procured From" badges inside Pricing & Stock card.

## 6. CSV Exports

Two download endpoints. All exports fetch with `Authorization: Bearer` token via a `downloadWithAuth()` helper in `web/lib/api.ts` — plain `<a href download>` was used initially but returned a 401 HTML page saved as `.htm`.

Both endpoints use CTEs for efficiency (single-pass aggregation across all products, not per-row correlated subqueries).

### `GET /api/analytics/health-export`
All products, sorted by supplier then avg monthly consumption desc.
Columns: `product_name`, `barcode`, `suppliers`, `avg_monthly_consumption`, `avg_daily_demand`, `predicted_daily_demand`, `last_7_day_avg`, `last_60_day_avg`, `volatility_pct`, `signal`.

### `GET /api/analytics/supplier-export?supplier=NAME`
Supplier-wise report. One row per product per supplier (JOIN on `product_supplier_mapping`, not a pipe-separated aggregate column) — keeps Excel filters clean.
Optional `?supplier=NAME` param filters to a single supplier; omit for all.
Columns: `supplier`, `product_name`, `barcode`, `mrp`, `purchase_price`, `current_stock`, `avg_monthly_consumption`, `predicted_daily_demand`, `signal`.
Sorted by supplier name, then avg monthly desc.

**UI** (Product Health page filter row):
- Product name/barcode search input
- Supplier autocomplete combobox — type to filter, X to clear, click to select. Populated from `GET /api/suppliers`.
- "Export Supplier" button — exports selected supplier only; falls back to full supplier report if none selected.
- "Full Export" button — all products, all signals.
