## 1. ML Strategy Scoping

Ran a full pre-B1 audit covering: what models make sense for this data, what business questions they answer, how to implement them, and what data gaps must be fixed first.

### Model landscape (beyond the roadmap's XGBoost mention)

| Layer | Models | Business question |
|---|---|---|
| Demand forecasting | LightGBM/XGBoost global, Croston/TSB (intermittent), Prophet/SARIMAX (topline), quantile regression | What to reorder, how much, by when? |
| Customer | RFM → KMeans/GMM, BG/NBD + Gamma-Gamma CLV, churn classifier | Who are VIPs? Who's about to lapse? |

### Implementation pattern decided

Batch-train offline (`ml/` folder) → write predictions to `app.ml_demand_predictions` → pipeline + dashboard read predictions like any other table. No live training UI for staff. MLflow for experiment tracking (port 5001). "Retrain Now" button in admin panel (subprocess, same pattern as pipeline trigger).

### Industry best practices locked in

- Walk-forward CV for time series, never random shuffle
- Quantile forecasts (P10/P50/P90) over point forecasts — inventory decisions need distribution
- Pinball loss at P90 as primary optimization objective (penalizes stockout > overstock)
- MASE/WAPE for catalog-wide metrics (MAPE blows up on low-volume SKUs)
- Champion/challenger: 4-week shadow before promoting new model
- Cold-start (SKU < 60 obs): use category-mean prior
- Separate model for intermittent demand (>50% zero days): Croston, not LightGBM
- Drift monitoring: KS test on feature distributions weekly; alert if MAPE degrades >20%
- No customer PII (mobile numbers) in model artifacts or MLflow logs

## 2. `derived.calendar_dim` + `scripts/seed_calendar.py`

New dimension table covering 2024-01-01 → 2029-12-31 (2,192 rows). Seeded once, refreshed annually.

**Script**: `scripts/seed_calendar.py`

Sources:
- Official Indian holidays via `holidays` Python library (`holidays.country_holidays('IN', years=yr)`) — covers Republic Day, Independence Day, Gandhi Jayanti, Diwali, Dussehra, Maha Shivaratri, Mahavir Jayanti, Guru Nanak Jayanti, Muharram, Buddha Purnima, etc.
- Hardcoded retail festivals with pre/post windows (library misses these or classifies them differently):

| Festival | Pre-window | Post-window |
|---|---|---|
| New Year | 2d | 1d |
| Pongal | 2d | 1d |
| Holi | 3d | 2d |
| Eid ul-Fitr | 2d | 1d |
| Eid ul-Adha | 2d | 1d |
| Ganesh Chaturthi | 3d | 2d |
| Onam | 3d | 2d |
| Navratri-Dussehra | 9d | 2d |
| Dhanteras | 3d | 1d |
| Diwali | 5d | 3d |
| Christmas | 3d | 1d |

Hardcoded through 2028; needs annual update after that. Script is idempotent (`ON CONFLICT DO UPDATE`).

**Columns**:
```
date, dow, is_weekend, month, quarter, week_of_year,
is_holiday, holiday_name,
is_festival_window, festival_name,
days_to_next_festival, days_since_last_festival
```

`days_to_next_festival` is the key ML feature: demand ramps 5–7 days before Diwali/Holi, not just on the peak day. Standard holiday binary flags miss this completely.

**Why not in `app.*`**: Calendar is a pure reference dimension, not application state. `derived.*` is correct placement. Unlike other derived tables, this one is NOT rebuilt by the pipeline (no entry in `sql/rebuild_derived/` order).

## 3. `02_product_daily_features.sql` — Three new columns

Added to `derived.product_daily_features`:

### `stockout_proxy` (BOOLEAN)

```sql
CASE
    WHEN quantity_sold = 0
     AND COALESCE(last_7_day_avg, 0) > 0.5
     AND purchase_quantity = 0
    THEN TRUE ELSE FALSE
END
```

Meaning: product was selling recently, nothing sold today, no restock arrived. High probability the shelf was empty rather than demand being zero.

**Critical ML implication**: rows where `stockout_proxy = TRUE` must be **excluded from training target** — they are censored observations (demand was suppressed by empty shelf, not by true zero demand). Training on them would bias all demand forecasts low, especially for slow/dead stock SKUs. Roughly 80K censored rows identified (~2.8% of ~2.85M total).

### `is_holiday`, `days_to_next_festival`, `days_since_last_festival`

Simple LEFT JOIN to `derived.calendar_dim` on `date`. Returns NULL if calendar not yet seeded (graceful degradation — pipeline won't fail on an unseeded machine).

### SQL restructure: CTE pattern

The original step 02 was a flat SELECT with window functions. To reference `last_7_day_avg` in the `stockout_proxy` CASE expression, the file was restructured as:

```sql
WITH windowed AS (
    -- all window functions computed here
    SELECT ... AVG() OVER (...) AS last_7_day_avg, ...
    FROM derived.product_daily_metrics
)
INSERT INTO derived.product_daily_features
SELECT
    ...,
    CASE WHEN quantity_sold = 0 AND last_7_day_avg > 0.5 AND ... THEN TRUE ... END AS stockout_proxy,
    c.is_holiday,
    c.days_to_next_festival
FROM windowed w
LEFT JOIN derived.calendar_dim c ON w.date = c.date;
```

PostgreSQL does not allow referencing window function aliases in a CASE expression in the same SELECT scope. CTE pulls them into an outer scope.

`ALTER TABLE ... ADD COLUMN IF NOT EXISTS` guards added before the TRUNCATE to handle schema migration on existing deployments without requiring a full table drop.

## 4. Migration 007 — `app.ml_demand_predictions`

```sql
CREATE TABLE IF NOT EXISTS app.ml_demand_predictions (
    date          DATE    NOT NULL,
    product_id    TEXT    NOT NULL,
    p10           NUMERIC NOT NULL,
    p50           NUMERIC NOT NULL,
    p90           NUMERIC NOT NULL,
    model_name    TEXT    NOT NULL,
    model_version TEXT    NOT NULL,
    predicted_at  TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (date, product_id)
);
```

Lives in `app.*` (not `derived.*`) so predictions survive pipeline rebuilds. When the model is retrained, the inference script bulk-replaces all rows. Two indexes: `(product_id, date)` for product-page queries, `(date)` for bulk daily reads.

Primary key is `(date, product_id)` — one active prediction per product per day. Champion/challenger complexity deferred; single active version at a time is sufficient for Phase B1.

**Integration plan** (for next session / B1 model code):
- Step 06 (`supplier_restock_recommendations`) will read `p90` from `app.ml_demand_predictions` as the safety-stock-aware demand estimate, falling back to the SQL WMA baseline when ML predictions are absent.
- Product drawer forecast chart: actual vs p50 with P10–P90 confidence band.
