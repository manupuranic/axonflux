# Demo Data Setup

## Why a Separate Demo Database

The production database contains real customer PII (mobile numbers, names) and actual business financials. The demo database uses only synthetic data generated by `scripts/generate_demo_data.py`.

The two databases are **identical in schema** — same Alembic migrations, same derived tables, same API. Only the data differs.

## Generating Demo Data

```bash
# Generates 6 months of synthetic FMCG data (Oct 2025 – Mar 2026)
python scripts/generate_demo_data.py
```

Output in `data/sample/`:
- `sales_itemwise/sales_itemwise_demo.csv` — ~90k item-level rows
- `sales_billwise/sales_billwise_demo.csv` — ~18k bill-level rows
- `purchase_billwise/purchase_billwise_demo.csv` — ~300 purchase orders

The generated data **deliberately includes realistic data quality issues**:
- Some products appear with two different barcodes (duplicate catalog entries — same product entered twice by different staff)
- Some barcodes map to slightly different names (human typo variants in item name field)
- Mixed name casing across entries ("SURF EXCEL 1KG" vs "Surf Excel 1 Kg")

These replicate real-world retail data problems and make the Product Entity Resolution feature (Phase B3) demonstrable on a live demo.

## Setting Up the Demo Database Locally

```bash
# 1. Create the demo database
createdb axonflux_demo

# 2. Create a .env.demo pointing to axonflux_demo
cp .env .env.demo
# Edit .env.demo: change  dbname=axonflux  →  dbname=axonflux_demo

# 3. Run migrations (swap env file temporarily)
cp .env .env.real && cp .env.demo .env
alembic upgrade head
cp .env.real .env  # restore

# 4. Copy sample CSVs to incoming
cp -r data/sample/sales_itemwise    data/incoming/
cp -r data/sample/sales_billwise    data/incoming/
cp -r data/sample/purchase_billwise data/incoming/
cp -r data/sample/supplier_master   data/incoming/

# 5. Run ingestion
PYTHONPATH=. python scripts/ingest_all.py

# 6. Rebuild all derived tables
PYTHONPATH=. python pipelines/weekly_pipeline.py

# 7. Create demo user
python scripts/create_admin.py
# Suggested: username=demo  password=demo123
```

## Switching Between Real and Demo

No code changes. Swap `.env` and restart uvicorn:

```bash
# Switch to demo
cp .env .env.real && cp .env.demo .env
# Restart uvicorn

# Switch back to real
cp .env.real .env
# Restart uvicorn
```

## Live Hosted Demo (Portfolio)

For a public URL you can share in job applications:

| Service | Role | Cost |
|---|---|---|
| [Neon](https://neon.tech) | PostgreSQL | Free (512 MB) |
| [Railway](https://railway.app) | FastAPI | ~$5/month or free credit |
| [Vercel](https://vercel.com) | Next.js | Free |

**Setup:**
1. Create Neon project → copy connection string
2. Run migrations + ingestion + pipeline against Neon (set `DATABASE_URL` env var)
3. Deploy FastAPI to Railway — point to Neon via env var
4. Deploy Next.js to Vercel — set `NEXT_PUBLIC_API_URL` to Railway URL

**Note:** The current auth has no read-only role. For a public demo, create a regular user with a known password and note it in your README. Avoid linking directly to the pipeline trigger or cash closure submit actions.
